# Feature Specification: LifeSoundtrack — save albums to a personal list

**Feature Branch**: `003-save-album-command`  
**Created**: 2026-04-23  
**Status**: Draft  
**Input**: User description: "Add a new command to the bot to allow users to save music albums that they want to listen to. Command usage example: `/album Abbey Road by The Beatles`. The command must accept any input from the user and should not validate or expect a specific format. Using an external API, the bot will search for the album metadata (e.g. title, artists, genres, year, and other helpful information). The bot will save the album data captured in a database. The bot will also save the user information in the database: name, username (e.g. Telegram username), external_id (e.g. Telegram user_id), source (e.g. Telegram), and other useful data points."  
**Amendment (2026-04-23)**: Disambiguation shows **at most two (2)** album candidates (plus an **Other** path). Each candidate control is **labeled** with **`ALBUM_TITLE | ARTIST (YEAR)`** (using available fields). **Other** delivers **help** to refine the query (e.g. try full album title, artist name, and year)—no album is saved for that request.  
**Amendment (2026-04-24)**: Metadata lookup uses a **fixed** **chain** of external catalogs in this order: **(1) Spotify**, **(2) iTunes (Apple)**, **(3) Last.fm**, **(4) MusicBrainz**. Each catalog has its own **on/off** **feature flag** set via **environment variables** (one variable per catalog; see planning for exact names and default values). A **disabled** catalog is **not** called. When a **enabled** catalog does **not** produce an outcome sufficient to complete the user’s request, or a **recoverable** failure occurs, the product **continues** to the **next** **enabled** catalog in order until a sufficient outcome is reached or the **chain** is exhausted. If **all** catalogs are **disabled** or the **chain** yields **no** acceptable result, the **no match** / **failure** behavior in User Story 2 still applies (no false success).  
**Amendment (2026-04-24 — equivalent labels)**: If the **at-most-two** **album** **candidates** that would be **offered** for disambiguation **all** **share** the **same** user-visible **label** in the form **`ALBUM_TITLE | ARTIST (YEAR)`** (per the same **display** **rules** for **title**, **primary** **artist(s)**, and **year**), the product **must not** ask the user to **choose** among **redundant** **lines**. The product **must** use the **highest-relevance** candidate, defined as the **first** in the **existing** **relevance** **ordering** among those **equivalents**, and **may** **complete** the **save** **without** a **disambiguation** **step** for that request (no **false** “pick between **identical** **labels**”).

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Save an album with a free-form message (Priority: P1)

A **listener** in a private chat with LifeSoundTrack sends a **save-album** command with **any** text they choose after the command (e.g. a title, a title plus artist, or informal wording). The product **resolves** that text to a real album using a **metadata lookup**, then **records** the album and the **user** so the item appears on the listener’s saved list and can be used in later product features.

**Why this priority**: This is the core value—turning a casual message into a structured, retrievable “want to listen” item without forcing the user to type a strict pattern.

**Independent Test**: In isolation, a user (or test double) issues the command with a single free-form string; the system either confirms success with a short, **non-technical** message or returns a **clear** failure that does not expose secrets. Data rows exist for the user and the album association when the flow succeeds.

**Acceptance Scenarios**:

1. **Given** a listener who is identified by the host platform, **When** they send the save-album command with a **non-empty** free-form text (any characters the platform allows, no required template), **and** the metadata lookup yields **exactly one** strong match, **Then** the product persists **album** fields returned or derived from the provider (e.g. title, primary artists, genre(s), release year) together with a link to the **user** who issued the command, and the user sees a **success** confirmation.
2. **Given** the same, **When** the free-form text is **empty** (only the command, no user query), **Then** the product responds with a **helpful** message explaining that a **search text** is required, and **does not** run metadata lookup or create a new stored album.
3. **Given** a successful save, **When** the same listener runs the command again on **another** free-form text, **Then** the product can store **another** album association for that user (v1 does **not** require merging duplicates unless specified later).

### User Story 1b - Choose among multiple search results (Priority: P1)

A **listener** issues the save command with a **vague** query (e.g. a single common album title) and the metadata source returns **several** plausible albums. For **different** releases (two or more **distinct** user-visible labels), the product **does not** guess **silently** which release to save: it **shows at most two** album lines **in relevance order** (and an **Other** path), each line **labeled** **`ALBUM_TITLE | ARTIST (YEAR)`**, so the listener can **pick the correct album** or ask for help. If **every** **candidate** that would be **offered** for that turn **shares** the **same** **label** (duplicate lines that mean the **same** album to the user), the product **must not** ask the user to **pick** between **redundant** **options**; it **applies** the **first** candidate in **relevance** **order** among those **equivalents** and **proceeds** like a **clear** **single** **match** for that case (**FR-009**). Only a **chosen** or **auto**-**resolved** album is stored (**Other** does not save an album when disambiguation is shown).

**Why this priority**: Ambiguous queries are common; auto-picking the **wrong** **distinct** **release** would break trust; asking the user to **choose** between **identical** **labels** wastes effort and **does not** add clarity. This story is on the **critical path** to a correct “save” for many real messages.

**Independent Test**: With a **stub** that returns at least two **different**-**label** albums for one query, the user **gets** a **disambiguation** step (unless they cancel), and after choosing, **that** release is saved. With a **stub** that returns **two** or **more** candidates **with** the **identical** **`ALBUM_TITLE | ARTIST (YEAR)`** **label** under the **display** **rules**, the user is **not** **prompted** to pick **between** **duplicates**; the product saves using the **highest**-**relevance** (first) **of** the **group**.

**Acceptance Scenarios**:

1. **Given** metadata lookup returns **two or more** relevant candidates and the command text is **non**-**empty**, **When** the product evaluates **user**-**visible** labels in the form **`ALBUM_TITLE | ARTIST (YEAR)`** (per **display** **rules**), **Then** it **merges** candidates that **share** the **same** **label** and **keeps** the **highest**-**relevance** entry for that label, defined as the **first** in the **existing** **relevance** **ordering** among **equivalents** **(see** **FR-009**). **If** there **are** then **two** **or** **more** **distinct** **labels** to **offer**, the product **presents** **at** **most** **two** **(2)** **lines** (the **top** **two** **distinct** **labels** **by** **relevance**), each **using** the **`ALBUM_TITLE | ARTIST (YEAR)`** form (year **omitted** when **unknown**), **plus** **Other**; the user **must** **pick** **one** line **or** **Other** before **any** **album** **row** is **written**. **If** there **is** **only** **one** **distinct** **label** (including the **all**-**duplicates** case), the product **does** **not** show **that** disambiguation **list**; it **saves** using the **kept** **candidate** **without** a **separate** **pick** **step** (see **FR-009**).
2. **Given** a disambiguation prompt, **When** the host platform supports **inline or reply buttons** for this interaction, **Then** the product **prefers** that UI: **two** controls match the two album lines (each **label** shows **`ALBUM_TITLE | ARTIST (YEAR)`**, not an option index), plus an **Other** control. **When** buttons are not available, **Then** the product lists **two** lines using the same **label** form and a clear **Other** line, and accepts the listener’s next message as a **choice** of **one** album line **or** **Other** (e.g. by index or keyword per plan), as in: query `/album Red`, many albums could match—`Red | Taylor Swift (2012)`, `Red | Gil Scott-Heron (1971)`, and **Other**; the user picks the first line and that album is saved.
3. **Given** a disambiguation prompt, **When** the user chooses **Other**, **Then** the product **does not** save an album for that request and sends a **short, helpful** message explaining how to **refine the query**—at minimum inviting the user to try **album full title**, **artist name**, and **release year** together (wording may vary; tone must stay **non-technical** and **encouraging**).
4. **Given** a single high-confidence match, **When** the provider returns one clear result (or a defined single-match path in planning), **Then** the product **may** skip the disambiguation list and **save** directly (per **FR-009** and plan).

### User Story 2 - When lookup fails, the user is not left guessing (Priority: P1)

The listener’s text may be ambiguous, the metadata service may be unavailable, or no **reasonable** match may exist. The product must respond in a **human-readable** way and must **not** leak internal keys, stack traces, or other users’ data.

**Why this priority**: Reliability and trust; failed saves are common when people type creatively or when services are down.

**Independent Test**: Trigger or simulate lookup failure, empty result set, and service error; every path shows a **short** user-facing message appropriate to the case, with **no** secret material in the reply. Verify persistence rules: **no orphan album record** is created for a **failed** end-to-end save (policy: do not store a “success” album row without a resolved match unless explicitly changed in a later spec).

**Acceptance Scenarios**:

1. **Given** a metadata lookup that returns **no** suitable match, **When** the user sent valid non-empty text, **Then** the user sees a **clear** “not found” (or similar) message and **no** new **album** row is created for a resolved release.
2. **Given** a **temporary** failure of the external metadata service, **When** the user issues the command, **Then** the user sees a **generic** failure or “try again” message, and the system does **not** claim success; **no** false confirmation.
3. **Given** the **first** one or more catalogs in the **metadata chain** are **disabled** or return a **recoverable** failure, **and** a **later** **enabled** catalog in the same chain **would** return a good match, **When** the user issues the command, **Then** the product can still **complete** a **successful** save (subject to **disambiguation** when needed); the user is **not** blocked solely because an **earlier** catalog was off or failed.

### User Story 3 - The listener’s identity is stored for the feature to work (Priority: P1)

The product must **create or update** a **listener profile** for the chat platform so that saved albums are tied to a **stable identity** (who saved what). This includes a **display name**, **platform handle** (where applicable), **external identifier** from the host, **source** of that identity, and any other **non-secret** fields needed for support and future features.

**Why this priority**: Without user persistence, the “my saved albums” story cannot exist across sessions.

**Independent Test**: After one successful save, a stored user record contains at minimum: **name** (or best available), **handle/username** (or null if absent on host), **external_id**, and **source**; re-invoking the command for the same platform user **updates** profile fields that may change (e.g. display name) per **FR** below.

**Acceptance Scenarios**:

1. **Given** a first-time listener who completes a **successful** save, **When** the flow finishes, **Then** a **user** (listener profile) record exists with **name**, **username** (if available), **external_id**, and **source** (e.g. the chat product name).
2. **Given** a returning listener whose **name** or **handle** changed on the host, **When** they save again, **Then** the product **updates** the stored profile for that **external_id** and **source** with current values, without duplicating a second “same person” record.

### User Story 4 - Operators and contributors can discover the feature (Priority: P2)

A **new team member** can read the product documentation, understand the command shape (free text), environment needs (e.g. credentials for **enabled** **metadata** **catalogs**, **per-catalog** **feature** **flags**, and **durable storage**), and how to test safely without affecting production data.

**Why this priority**: Reduces time-to-first-successful integration and support burden.

**Independent Test**: Follow the **feature quickstart** (to be created in planning) with test credentials: command works end-to-end in a **sandbox** environment.

**Acceptance Scenarios**:

1. **Given** a reader of the runbook, **When** they configure required secrets and run the app, **Then** they can complete one successful **save** path and one **intentional failure** path (e.g. no match) with documented expectations.

### Edge Cases

- **Extremely long** free-form text: the product may **truncate** for the provider request or return a “too long” user hint; behavior must be **documented** and must not crash the process.
- **Empty** command text: see Story 1; no metadata lookup, no new album.
- **Special characters, emoji, multiple languages** in the query: still **one** free-form string; no format validation beyond platform limits; lookup **may** still fail—Story 2 applies.
- **Metadata provider returns multiple** candidates: see **User Story 1b**; **candidates** that **share** the **same** **`ALBUM_TITLE | ARTIST (YEAR)`** **label** **do** **not** each get a **separate** **user** **line**—the product **keeps** the **highest**-**relevance** **(first)** **among** **equivalents** (see **FR-009**). **If** **two** **or** **more** **distinct** **labels** **remain** **(up** **to** **two** **offered)**, the product shows those **lines**, an **Other** path, **then** requires an **explicit** **pick** or **Other**; **if** only **one** **distinct** **label** **remains** after **collapse** (for example, **all**-**duplicate** **labels**), **no** **pairwise** **pick** is **required**. **Session** timeout, **duplicate** **inbound** **messages** during disambiguation, and cancel semantics are **plan-owned** but must be **user-safe** (no partial save of a **user**-**unconfirmed** **distinct** **release** when a **list** is **shown**; **Other** must not write an album row).
- **Concurrent** commands from the same user: v1 need not guarantee global ordering; each command is processed independently; duplicate rows possible if the user double-taps.
- **Host platform** not Telegram: the **domain** command and payload are still “save album with free text”; **source** and **field mapping** are adapter-specific. Telegram is the first supported host unless planning narrows further.
- **All** **four** **metadata** feature flags off (or **no** catalog **enabled**): the product does **not** run a meaningful lookup; the user gets a **clear** failure (e.g. configuration or availability) and **no** new album row for a “successful” match—consistent with **FR-006** and the **all-disabled** case below.
- **Chained** **catalogs**: if **one** **enabled** source returns **no** candidates and the product **moves** to the **next** **enabled** source, the **user** should see **one** coherent flow (not one error per catalog); how **candidates** from different catalogs are **merged** or **prioritized** for **disambiguation** is **plan-owned** but must still satisfy **FR-009** when multiple candidates exist.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The product MUST expose a **save-album** interaction (see platform mapping in the implementation plan) that accepts a **single** user-supplied string **as-is**; the product MUST **not** require a specific syntax (e.g. `Title | Artist`) for v1.
- **FR-002**: For a **non-empty** query string, the product MUST run **outbound** metadata lookup through a **fixed-order chain** of **four** **named** external catalogs: **(1) Spotify**, **(2) iTunes (Apple)**, **(3) Last.fm**, **(4) MusicBrainz**—in that order only. The product MUST expose **one** **environment** **variable** (boolean-style **feature flag**) per catalog so operators can **independently** **enable** or **disable** each integration **without** code changes. **When** a catalog’s flag is **off**, the product MUST **not** invoke that catalog. **When** a catalog’s flag is **on**, the product MUST invoke it as part of the chain **after** all **higher-priority** **enabled** catalogs have been **considered** according to the **chain** **rules** in this requirement. The product MUST obtain **one or more** **structured** album **candidates** including at least **title** and **primary artist(s)** per candidate, and SHOULD obtain **genres** and **release year** when a **catalog** supplies them, plus any other fields agreed in planning as **user-visible** or **analytically** useful. The product MUST **continue** the chain when: a **higher** **priority** **enabled** catalog is **skipped** because its flag is **off**; a **higher** **priority** **enabled** catalog **fails** in a way that is **treated** as **fallthrough**; or a **higher** **priority** **enabled** catalog returns **no** suitable candidates—each as defined in planning—until a **sufficient** outcome exists or **all** **enabled** catalogs in the **order** are exhausted. The product MUST then apply **FR-009** when **metadata** returns **candidates** that **imply** **multi**-**outcome** **handling** (more **than** **one** **raw** **match** **or** **chain**-**merged** **sets**), including **label** **collapse** and **disambiguation** (see **FR-009**). The product MUST **not** require a **disabled** catalog to **succeed**. If **no** catalog is **enabled** (all flags **off**), the product MUST **not** claim a successful **metadata** **resolution**; user-facing and logging behavior MUST align with **FR-006** and **all-disabled** **edge** **cases**. Exact **variable** **names**, **default** flag **values** (e.g. all **on** in production templates), and **parsing** rules (e.g. `true`/`false`, `1`/`0`) are **documented** in the **plan** and **runbook**.
- **FR-003**: On **successful** resolution of a match, the product MUST **persist** the album (or a normalized representation of the provider’s result) in application storage so it can be listed or queried later, and MUST **link** that save to the **listener** who issued the command.
- **FR-004**: The product MUST **create or update** a **listener profile** per **(source, external_id)** (or the equivalent composite key) containing at least: **name** (display name or best available), **username** / **handle** (when the host provides it; nullable otherwise), **external_id**, and **source**; additional non-secret **profile** fields are allowed as documented in planning.
- **FR-005**: If **metadata** lookup does **not** yield an acceptable match, the product MUST **not** create a new stored **album** record for a “successful” discography entry; the user receives an explanatory message (per Story 2).
- **FR-006**: If **metadata** lookup fails for **infrastructure** reasons (network, rate limit, server error), the product MUST return a user-appropriate error and MUST **not** state that the album was saved.
- **FR-007**: User-visible and log output for this feature MUST follow existing **safety and tone** rules for the product: **no** API keys, **no** database connection strings, **no** other users’ PII, and **no** full raw provider payloads in the default user message (summaries or confirmations only).
- **FR-008**: The **first** host adapter will map the product’s **save-album** command to the **Telegram** bot interface (e.g. `/album …` with trailing text) unless a subsequent amendment narrows scope; the **core** name for the feature remains **host-neutral** in domain documentation.
- **FR-009**: When metadata lookup returns **more than one** relevant **raw** **candidate**, the product MUST first **map** each **candidate** to a **user**-**visible** **label** **`ALBUM_TITLE | ARTIST (YEAR)`** (using the best available title, primary artist(s), and year; *year* omitted when unknown—see planning for display rules) and **collapse** any **candidates** that **share** the **same** **label**, **keeping** **only** the **highest**-**relevance** **candidate** for each **label**—**defined** as the **first** in the **relevance** **ordering** when **raw** **rows** with that **label** are **compared** **(first**-**wins** **among** **equivalents**)**. **If** **after** **this** **step** there **is** **exactly** **one** **remaining** **distinct** **label** (for example, **all** **raw** **candidates** were **duplicates** of the **same** **label**), the product MUST **not** show a **disambiguation** **list** and MUST **proceed** to **save** the **kept** **album** (same **outcome** **class** as a **strong** **single** **match** for **user** **clarity**), **without** **requiring** a **user** **pick** **between** **redundant** **options**. **If** **after** **collapse** there **remain** **two** **or** **more** **distinct** **labels**, the product MUST **not** persist any album for that request until the user **selects** one of **at** **most** **two** **(2)** **offered** **album** **lines** **or** **chooses** **Other**. The product MUST present **at** **most** **two** **(2)** **album** **lines** from the **set** of **distinct** **labels** **(when** **more** **than** **two** **distinct** **labels** **exist**, **only** the **top** **two** **by** **relevance** **are** **offered**)**. The product MUST offer an **Other** path **only** when **that** **true** **disambiguation** **list** **(two** **distinct** **labels**)** is **shown**; **it** is **not** **required** to **show** **Other** when **the** **product** **skips** **the** **list** **because** **only** **one** **distinct** **label** **remains** after **collapse**. Each **shown** **album** **line** MUST be **identified in the UI** with the **`ALBUM_TITLE | ARTIST (YEAR)`** form. The product MUST **not** use bare option indices as the **only** user-visible label for an album (indices may still exist for text fallback). The product MUST use **platform-native chooser controls** (e.g. **inline / reply buttons**) when the host supports them; **otherwise** the product MUST use a **list** in chat with the same **label** form and a clear **Other** line when **disambiguation** **applies**, and treat the user’s next message (or the platform’s callback) as the **selection** of **one** album or **Other**. If the user selects **Other** on a **disambiguation** **prompt**, the product MUST send **guidance** to refine the query (including **album full title**, **artist name**, and **year** in the help text) and MUST **not** create a new stored album for that request. A **single** **unambiguous** **raw** **result** (or a **path** the **plan** **treats** as **single**-**match**) **MAY** proceed **without** a **disambiguation** **step** **(including** when **collapse** **reduced** **many** **raw** **duplicates** to **one** **label**).**

### Non-functional and product quality *(should align with project constitution)*

- **UX consistency**: Confirmation and error messages for **save album** use the same tone and product naming conventions as the rest of LifeSoundTrack (e.g. **LifeSoundtrack** where appropriate); **help**-style text should be updated so listeners discover **/album** (or equivalent) alongside **start** / **help** / **ping** when those exist.
- **Testing**: Automated tests SHOULD cover: empty query, at least one **happy** path (with a **recorded** or **stubbed** provider), **multi-match disambiguation** (stub returns ≥2 **distinct**-**label** **options** → user selects one of at most two labeled lines → correct record saved), **equivalent**-**label** **collapse** (stub returns **≥2** **raw** **candidates** **with** the **identical** **label** → **no** **disambiguation** **list**; **first**-**relevance** **candidate** is **saved**), **Other** (no save, help text present), no-match, and **provider error**; **at least one** path where an **earlier** **chain** **step** is **disabled** or **errors** and a **later** **step** still **succeeds**; and **at least one** path where **all** **metadata** **flags** are **off** (or no **step** is **enabled**), expecting **no** false **success**; persistence SHOULD be testable with an **in-memory** or test database in CI when the repository’s testing strategy allows.
- **Linting / static analysis**: New code packages MUST follow repository conventions; any **generated** or **migrated** schema MUST be **reviewed** in PR.
- **Logging and monitoring**: Operations SHOULD log **command outcome** and **error class** (e.g. no match vs provider error) without secret headers or user tokens; optional correlation IDs for support.
- **Performance**: A typical save attempt SHOULD complete from the user’s perspective within a **documented** target (e.g. under 15 seconds) when the provider and database are healthy; if slower, a “still working” pattern is a **P2** enhancement.
- **Containerization / local dev**: This feature is expected to add or modify **persistence**; the plan MUST update or add **Dockerfile** and **Docker Compose** (or the repo’s standard equivalent) so contributors can run the bot and database together locally, per constitution **VIII** where applicable.
- **Decommissioning**: If the feature is later replaced, remove or archive **/album** copy, runbooks, and unused tables/migrations in the same **release window** and update **constitution/IX**-relevant links.

### Key Entities

- **Listener (user profile)**: Represents a person using the product through a given **source**; keyed by **source** + **external_id**; stores **name**, **username** (optional), and other **non-secret** profile fields needed for attribution and support.
- **Saved album (list item)**: Represents one **catalog album** (or provider-normalized set of metadata) **saved by** a listener; holds provider-derived fields (title, artists, year, genre(s), **provider identifiers** for dedup or re-fetch) and a **link** to the **listener**; may include a **captured** original query string for user reference.
- **Metadata lookup (conceptual)**: Not a user-visible entity, but an **ordered** **chain** of **up to four** external **catalogs**—**Spotify** → **iTunes** → **Last.fm** → **MusicBrainz**—each of which can be **turned** **on** or **off** with an **environment** **feature** **flag**; the plan binds **unified** **search** **semantics**, **fallthrough** (skip disabled, try next on non-success per rules), and **mocks** for **tests** to this **port**.
- **Disambiguation (pending choice)**: An ephemeral interaction state: **(listener, up to two distinct album lines by user-visible label when a list is shown, an “Other” branch when that list is shown, relevance order, optional message/thread identifiers)**. **Same**-**label** **raw** **duplicates** are **merged** per **FR-009** before a **list** is **offered**; only **distinct** **labels** count toward the **at-most-two** cap. **v1** uses **PostgreSQL** (or in-process state for single-process dev) only—**no** Redis. Must remain **valid** long enough to accept one user **pick**; timeout and cancel are plan-owned.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: In **structured tests** (automated and/or a scripted **sandbox**), **100%** of **happy-path** invocations with **non-empty** text result in a **single** new **saved album** record associated with the **correct** listener, with **at least** title and **primary** artist(s) populated when the provider returns them—**including** paths where the user **first** disambiguates among **at most two** **distinct**-**by**-**label** **options** (and **not** when choosing **Other**) **then** saves, and **including** the **all**-**duplicate**-**label** case (no disambiguation **list**; **kept** **first**-**relevance** **row** **saved**), per **SC-007** and **FR-009**.
- **SC-002**: In **100%** of **intentional** no-match and provider-error cases in the test suite, the user receives a **non-success** message and **durable storage** does not show a new **“successful”** album save for a resolved discography object for that attempt.
- **SC-003**: In **100%** of reviewed **log** and **user message** samples in acceptance review, there is **no** occurrence of **API keys**, **database credentials**, or **other users’** private identifiers in user-visible or default **Info-level** log lines for this feature.
- **SC-004**: A **new contributor** (or a reviewer) following the post-planning quickstart can complete (a) a successful save and (b) a no-match case in **one** work session, **without** reading application source to guess configuration—evidence: checklist sign-off or PR review.
- **SC-005**: In **structured tests** for the **disambiguation** path, when the user selects **Other**, **100%** of cases **do not** create a new **saved album** record and the user-visible message **includes** guidance to refine the query using **full album title**, **artist name**, and **year** (or equivalent non-technical wording).
- **SC-006**: In **structured tests** that **disable** or **force** **failure** on **one** or **more** **earlier** **steps** in the **metadata** **chain** while a **later** **enabled** **step** **can** still **resolve** a **match**, **100%** of such cases either **save** a **correct** **album** (when a **match** **exists**) or return a **non-success** user outcome **without** a **false** “saved” **claim**; **100%** of cases with **no** **enabled** **catalog** **step** (all **flags** **off** or **equivalent** **misconfiguration**) yield a **non-success** user outcome and **no** new **“successful”** **album** **row** for a **resolved** **release**.
- **SC-007**: In **structured tests** where **lookup** **yields** **at** **least** **two** **candidates** **with** the **identical** **user**-**visible** **`ALBUM_TITLE | ARTIST (YEAR)`** **label** under the **display** **rules**, **100%** of such cases **do** **not** show a **disambiguation** **prompt** **with** **duplicate** **lines**; the **listener** **completes** a **successful** **save** **for** **that** **query** using the **highest**-**relevance** (first) **match** **among** **those** **equivalents** (see **FR-009**).

## Assumptions

- **Private chat** remains the default context, consistent with feature **001**; group chat is out of scope unless a later spec extends it.
- The **metadata** **stack** is **defined** in this spec as the **chain** and **flags** in **FR-002**; **rate** **limits**, **credentials**, **per-catalog** field **mapping**, and **merge** rules across catalogs are **detailed in the implementation plan** and **runbook**.
- **Default** **feature** **flag** **values** (e.g. all catalogs **on** in **standard** **deploys**) are **stated in planning** so operators can **disable** **troublesome** or **unneeded** **integrations** without a **code** **change**.
- **v1** does **not** require **de-duplication** of the same discography item across multiple saves by the same user; saving twice may create two rows. **Deduplication** and “already in your list” behavior can be a follow-up.
- **Currency / locale** of release year and genre labels follow **provider** data; the product may display them **as provided** in English or raw form for v1.
- **List / browse** my saved albums is **implied** as a future or companion feature; **v1** focuses on **add**; a minimal **acknowledgment** line in the success message is enough to prove persistence (e.g. “Saved *Album* (Year)”).
- **PII and retention** for listener profiles and saved rows follow the team’s default **privacy** policy and region requirements; the plan may add **deletion** or **export** stories later.
- **Disambiguation list size**: The product shows **at most two (2)** **distinct**-**by**-**label** album **lines** per prompt **(when** a **list** **is** **shown)**, **plus** an **Other** path **for** that **list**; with **one** **distinct** **label** after **merging** **equivalent** **rows** (**FR-009**), a **user** **pick** is **not** **required** and an **Other** line **is** **not** **required** for **that** **turn**; with **one** **strong** **raw** **result**, a direct save **may** apply. *This supersedes prior “up to three (3) options” phrasing: the cap for **distinct** album **lines** is two; the **Other** path is in addition, not a third album row when disambiguation **applies**.*
- **No Redis (v1)**: Pending disambiguation state is **not** stored in Redis; use **PostgreSQL** (durable) and/or **in-process** for single-node dev, per **Clarifications** (Session 2026-04-23).

## Clarifications

### Session 2026-04-24

- **Equivalent**-**label** **duplicates** **(regression)**: The product must not prompt the user to **choose** between **two** **or** **more** **options** that **use** the **identical** **`ALBUM_TITLE | ARTIST (YEAR)`** **label**; it **merges** **per** **FR-009** and **proceeds** with the **highest**-**relevance** **(first)** **row**. This **is** the **authoritative** **fix** for **redundant** **disambiguation** **lists** (see **SC-007**).

### Session 2026-04-23

- **Q:** When metadata lookup returns **several** matching albums, how should the user select one? **A:** The product must offer **at most two** album **choices** **ordered by relevance** when **two** **or** **more** **distinct** **by**-**label** **releases** **need** **a** **pick** (if more than two **distinct** **labels** match, show the **top** **two** **distinct** **labels**), each **labeled** **`ALBUM_TITLE | ARTIST (YEAR)`**, plus an **Other** path **for** **that** **list**. **Other** does **not** save an album; it sends **help** to refine the query (e.g. include full title, artist, and year). Use **platform buttons** (inline/reply) when the host supports them; **otherwise** use a text list with the same **label** form and a clear **Other** line. **Persist** only a **user**-**selected** **line** **when** a **list** **is** **shown**; **no** **silent** **auto**-**pick** **between** **different** **releases**. **Exception** **(amended** **2026-04-24)**: **see** **Amendment** **(equivalent** **labels)** **and** **FR-009**—**candidates** **that** **share** **one** **user**-**visible** **label** **collapse** to the **first** **by** **relevance** **without** a **pick** **step**. *Amended 2026-04-23: supersedes prior “up to three” answer; 2026-04-24: **distinct**-**by**-**label** cap **and** **duplicate**-**label** **auto**-**resolve**.*

- **Q:** Should disambiguation state use **Redis**? **A:** **No.** **v1** does **not** use Redis. Pending choices are stored in **PostgreSQL** (durable, multi-replica) and/or **in-process** memory for **single-instance** local dev only.
